# Отчет по лабораторной работе: Исследование отказоустойчивости вычислительных систем

## Введение

В данной лабораторной работе проводится анализ производительности коммуникационных каналов в вычислительных системах. Цель работы – измерить пропускную способность (bandwidth) и задержку (latency) на различных уровнях коммуникационной иерархии: контроллер памяти, внутрисистемная шина и сетевой адаптер. Для этого используется MPI-программа с асинхронными обменами и параметризацией размера сообщения.

## Теоретические основы

### Программный комплекс

Программный комплекс представляет собой одну MPI-программу, которая включает в себя:
- Таймеры для измерения времени выполнения операций
- Цикл измерений для получения статистики
- Асинхронные обмены с использованием `MPI_Isend` и `MPI_Irecv`
- Параметризацию размера сообщения для анализа зависимости производительности от объема передаваемых данных

### Асинхронные операции в MPI

`MPI_Isend` и `MPI_Irecv` – это асинхронные (неблокирующие) операции обмена в MPI. Они являются неблокирующими версиями стандартных операций `MPI_Send` и `MPI_Recv`. Ключевое отличие заключается в том, что они возвращают управление немедленно, не дожидаясь завершения передачи данных. Это позволяет программе продолжить выполнение других операций во время передачи данных, что повышает эффективность использования вычислительных ресурсов.

### Архитектурные особенности вычислительных систем

#### SMP (Symmetric Multiprocessing) – Симметричная многопроцессорная обработка

| Характеристика       | Описание                                                                                                 |
| -------------------- | -------------------------------------------------------------------------------------------------------- |
| **Архитектура**      | Все процессоры подключены к **единой общей шине памяти** (обычно FSB – Front Side Bus)                   |
| **Доступ к памяти**  | **Однородный (UMA)**: все процессоры имеют одинаковое время доступа ко всей оперативной памяти           |
| **Память**           | Единый пул памяти, физически разделённый на модули, но логически единый                                  |
| **Масштабируемость** | Ограниченная: при увеличении числа процессоров шина памяти становится «узким местом» (конфликты за шину) |
| **Пример из лабы**   | Кластер Pine: платформа Intel SR2520SAF, процессоры Xeon E5420, шина **FSB**, 8 ГБ DDR2                  |

#### NUMA (Non-Uniform Memory Access) – Неоднородный доступ к памяти

| Характеристика       | Описание                                                                                              |
| -------------------- | ----------------------------------------------------------------------------------------------------- |
| **Архитектура**      | Каждый процессор имеет **собственный контроллер памяти** и локальную память; процессоры соединены высокоскоростной шиной (QPI) |
| **Доступ к памяти**  | **Неоднородный**: доступ к «своей» локальной памяти быстрее, чем к «чужой» удалённой памяти другого процессора |
| **Память**           | Физически распределённая: часть памяти «привязана» к процессору 0, часть – к процессору 1             |
| **Масштабируемость** | Высокая: отсутствует единая шина памяти, каждый процессор обслуживает свою память                      |
| **Пример из лабы**   | Кластер Oak: платформа Intel S5520UR, процессоры Xeon E5620, шина **QPI**, 24 ГБ DDR3                 |

## Исследуемые уровни коммуникации

Согласно методичке, необходимо исследовать три уровня коммуникации:

| Уровень                     | Что измеряем                                                                            | Где запускать на Pine                    | Привязка процессов                                                            |
| --------------------------- | --------------------------------------------------------------------------------------- | ---------------------------------------- | ----------------------------------------------------------------------------- |
| **1. Контроллер памяти**    | Доступ к локальной оперативной памяти                                                   | 1 узел, оба процесса на **одном сокете** | `numactl --physcpubind=0,1 --membind=0`                                       |
| **2. Внутрисистемная шина** | Обмен между сокетами через шину (на Pine – скорее всего **не QPI**, аналогичная шина) | 1 узел, процессы на **разных сокетах**   | `numactl --physcpubind=0 --membind=0` + `numactl --physcpubind=4 --membind=1` |
| **3. Сетевой адаптер**      | Межузловая коммуникация через **Gigabit Ethernet**                                      | **2 разных узла** кластера Pine          | Без привязки или `--bind-to core`                                             |

## Используемый код

### main.c

```c
#include <mpi.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>

#define ITERATIONS      1000    
#define WARMUP          20      
#define MIN_SIZE        1       
#define MAX_SIZE_MB     16      

int main(int argc, char** argv) {
    int rank, size;
    MPI_Init(&argc, &argv);
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &size);

    if (size != 2) {
        if (rank == 0) {
            fprintf(stderr, "Ошибка: программа требует запуска с 2 процессами (mpirun -np 2)\n");
        }
        MPI_Finalize();
        return 1;
    }

    int sizes[25];
    int num_sizes = 0;
    for (int exp = 0; exp <= 24; exp++) {
        int sz = 1 << exp;  
        if (sz <= MAX_SIZE_MB * 1024 * 1024) {
            sizes[num_sizes++] = sz;
        }
    }

    if (rank == 0) {
        printf("# Size(bytes)\tTime(sec)\tBandwidth(MB/s)\tLatency(us)\n");
        printf("# -----------------------------------------------------------\n");
        fflush(stdout);
    }

    for (int s = 0; s < num_sizes; s++) {
        int msg_size = sizes[s];
        char *sendbuf = (char*)malloc(msg_size);
        char *recvbuf = (char*)malloc(msg_size);
        
        if (!sendbuf || !recvbuf) {
            fprintf(stderr, "rank %d: ошибка выделения памяти для %d байт\n", rank, msg_size);
            MPI_Abort(MPI_COMM_WORLD, 1);
        }

        memset(sendbuf, (rank == 0) ? 0xAA : 0x55, msg_size);

        for (int i = 0; i < WARMUP; i++) {
            MPI_Request req[2];
            if (rank == 0) {
                MPI_Isend(sendbuf, msg_size, MPI_BYTE, 1, 0, MPI_COMM_WORLD, &req[0]);
                MPI_Irecv(recvbuf, msg_size, MPI_BYTE, 1, 0, MPI_COMM_WORLD, &req[1]);
            } else {
                MPI_Irecv(recvbuf, msg_size, MPI_BYTE, 0, 0, MPI_COMM_WORLD, &req[0]);
                MPI_Isend(sendbuf, msg_size, MPI_BYTE, 0, 0, MPI_COMM_WORLD, &req[1]);
            }
            MPI_Waitall(2, req, MPI_STATUSES_IGNORE);
        }

        MPI_Barrier(MPI_COMM_WORLD);
        double start = MPI_Wtime();

        for (int i = 0; i < ITERATIONS; i++) {
            MPI_Request req[2];
            if (rank == 0) {
                MPI_Isend(sendbuf, msg_size, MPI_BYTE, 1, 0, MPI_COMM_WORLD, &req[0]);
                MPI_Irecv(recvbuf, msg_size, MPI_BYTE, 1, 0, MPI_COMM_WORLD, &req[1]);
            } else {
                MPI_Irecv(recvbuf, msg_size, MPI_BYTE, 0, 0, MPI_COMM_WORLD, &req[0]);
                MPI_Isend(sendbuf, msg_size, MPI_BYTE, 0, 0, MPI_COMM_WORLD, &req[1]);
            }
            MPI_Waitall(2, req, MPI_STATUSES_IGNORE);
        }

        MPI_Barrier(MPI_COMM_WORLD);
        double end = MPI_Wtime();
        double total_time = end - start;
        double avg_time = total_time / ITERATIONS;

        if (rank == 0) {
            double bandwidth_mbs = (msg_size * 1.0) / (avg_time * 1024 * 1024); 
            double latency_us = avg_time * 1e6;                                
            
            printf("%d\t\t%.9f\t%.3f\t\t%.3f\n", 
                   msg_size, avg_time, bandwidth_mbs, latency_us);
            fflush(stdout);
        }

        free(sendbuf);
        free(recvbuf);
    }

    MPI_Finalize();
    return 0;
}
```

### Объяснение кода

Программа представляет собой MPI-приложение для измерения пропускной способности и задержки между процессами. Она состоит из следующих ключевых компонентов:

1. **Инициализация**: Используется `MPI_Init` для инициализации среды MPI, а затем определяются ранг (`rank`) и размер (`size`) коммуникатора `MPI_COMM_WORLD`. Ранг определяет уникальный идентификатор процесса (0 или 1), а размер - общее количество процессов (в нашем случае всегда 2).

2. **Проверка количества процессов**: Программа требует ровно 2 процесса для корректной работы. Если запущено другое количество процессов, выводится сообщение об ошибке и программа завершается.

3. **Генерация размеров сообщений**: Создается массив `sizes[]`, содержащий 25 размеров сообщений от 1 байта до 16 МБ, используя степени двойки (1, 2, 4, 8, ..., 16777216 байт). Это позволяет протестировать производительность на широком диапазоне размеров сообщений.

4. **Цикл измерений**: Для каждого размера сообщения выполняются следующие шаги:
   - Выделение памяти под буферы отправки (`sendbuf`) и приема (`recvbuf`) с помощью `malloc`
   - Проверка успешности выделения памяти для предотвращения ошибок
   - Заполнение буферов тестовыми данными (0xAA для процесса 0, 0x55 для процесса 1) для обеспечения надежности передачи
   
5. **Прогрев системы**: Перед основными измерениями выполняется 20 итераций прогрева (`WARMUP`), чтобы:
   - Убедиться, что все буферы выделены
   - Привести систему в стабильное состояние
   - Позволить планировщику настроить распределение ресурсов
   - Исключить влияние начальных задержек на результаты измерений

6. **Асинхронные обмены**: Используются неблокирующие функции `MPI_Isend` и `MPI_Irecv`:
   - Процесс 0 отправляет данные процессу 1 и одновременно принимает данные от него
   - Процесс 1 принимает данные от процесса 0 и одновременно отправляет ему данные
   - Это позволяет максимально эффективно использовать коммуникационный канал
   - `MPI_Request req[2]` используется для отслеживания состояния операций

7. **Синхронизация**: 
   - `MPI_Barrier` синхронизирует оба процесса перед началом и после окончания измерений
   - `MPI_Waitall` ожидает завершения всех асинхронных операций перед продолжением

8. **Измерение времени**: 
   - Используется `MPI_Wtime()` для получения точного времени до и после цикла измерений
   - Выполняется 1000 итераций (`ITERATIONS`) для получения статистически значимых результатов
   - Общее время делится на количество итераций для получения среднего времени одной передачи

9. **Расчет метрик**:
   - Пропускная способность вычисляется как `размер_сообщения / (среднее_время * 1024 * 1024)` в МБ/с
   - Задержка (латентность) вычисляется как `среднее_время * 1e6` в микросекундах
   - Эти метрики позволяют оценить как скорость передачи данных, так и задержку каждой операции

10. **Вывод результатов**: Только процесс с рангом 0 выводит результаты в формате: размер сообщения, среднее время, пропускная способность и задержка. Это предотвращает дублирование данных в выводе.

11. **Освобождение памяти**: После завершения измерений для каждого размера сообщения освобождаются выделенные буферы с помощью `free()`.

12. **Завершение**: Программа корректно завершает работу с помощью `MPI_Finalize()`.

### Объяснение разделения по уровням для уровня 1

Для измерения пропускной способности на уровне контроллера памяти мы использовали следующие подходы:

1. **Размещение процессов**: Оба процесса были размещены на одном сокете (сокет 0) с помощью команды `numactl --cpunodebind=0 --membind=0`, что обеспечивает максимальную близость к одному и тому же контроллеру памяти.

2. **Привязка к NUMA-узлу**: Параметр `--membind=0` гарантирует, что выделенная память будет размещена на NUMA-узле 0, что минимизирует задержки доступа к памяти.

3. **SLURM параметры**: Использование `--nodes=1 --ntasks-per-node=2` гарантирует, что оба процесса будут запущены на одном узле, что исключает межузловую коммуникацию.

4. **Тип коммуникации**: На этом уровне фактически происходит взаимодействие через общую память внутри одного сокета, что обеспечивает наивысшую пропускную способность и минимальную задержку.

### Объяснение разделения по уровням для уровня 2

Для измерения пропускной способности на уровне внутрисистемной шины мы использовали следующие подходы:

1. **Размещение процессов**: Параметр `--map-by socket:PE=1` гарантирует, что два процесса будут размещены на разных сокетах (по одному на каждый сокет), что позволяет измерить производительность межсокетного взаимодействия.

2. **Привязка к ядрам**: Параметр `--bind-to core` обеспечивает привязку процессов к конкретным ядрам, что улучшает предсказуемость и воспроизводимость результатов.

3. **SLURM параметры**: Использование `--nodes=1 --ntasks-per-node=2` гарантирует, что оба процесса будут запущены на одном узле, исключая влияние сетевой коммуникации.

4. **Тип коммуникации**: На этом уровне происходит обмен данными между процессами на разных сокетах одного узла через внутреннюю шину (FSB на системах Pine), что дает промежуточные характеристики между уровнем контроллера памяти и сетевым адаптером.

### Объяснение разделения по уровням для уровня 3

Для измерения пропускной способности на уровне сетевого адаптера мы использовали следующие подходы:

1. **Размещение процессов**: Параметр `--nodes=2 --ntasks-per-node=1` гарантирует, что каждый процесс будет запущен на отдельном физическом узле, что обеспечивает максимальную удаленность процессов друг от друга.

2. **SLURM параметры**: Использование двух узлов исключает любые внутренние коммуникации внутри узла и заставляет все обмены происходить через сеть.

3. **Тип коммуникации**: На этом уровне происходит межузловая коммуникация через сетевой адаптер Gigabit Ethernet, что дает наихудшие характеристики по сравнению с другими уровнями.

4. **Отсутствие привязки**: В этом случае не используется специальная привязка процессов к конкретным сокетам или ядрам, так как каждый процесс работает на отдельном узле.

## Анализ результатов

### Сравнительный анализ пропускной способности

Из полученных данных видно, что пропускная способность различается на каждом уровне:

1. **Уровень 1 (Контроллер памяти)**: На этом уровне наблюдается самая высокая пропускная способность. Для больших сообщений (8-16 МБ) она достигает около 16 ГБ/с. Это объясняется тем, что оба процесса находятся на одном сокете и используют общую память, что минимизирует задержки.

2. **Уровень 2 (Внутрисистемная шина)**: Пропускная способность на этом уровне ниже, чем на первом уровне, но все еще достаточно высока. Для больших сообщений она составляет около 10 ГБ/с. Разница объясняется тем, что процессы находятся на разных сокетах и должны использовать внутреннюю шину для обмена данными.

3. **Уровень 3 (Сетевой адаптер)**: На этом уровне пропускная способность значительно ниже, чем на предыдущих уровнях. Для больших сообщений она составляет около 140 МБ/с. Это связано с тем, что обмен происходит через сетевой адаптер Gigabit Ethernet, который имеет ограниченную пропускную способность по сравнению с внутренними шинами.

### Анализ задержек

Задержки также различаются на разных уровнях:

1. **Уровень 1**: Минимальные задержки, особенно для маленьких сообщений. Например, для 1 байта задержка составляет всего 0.39 мкс.

2. **Уровень 2**: Задержки выше, чем на первом уровне, но все еще относительно низкие. Для 1 байта задержка составляет 0.711 мкс.

3. **Уровень 3**: Наибольшие задержки, особенно заметные для маленьких сообщений. Для 1 байта задержка составляет 57.318 мкс.

### Зависимость от размера сообщения

На всех уровнях наблюдается зависимость характеристик от размера сообщения:

- Для маленьких сообщений (до нескольких килобайт) пропускная способность невысока, так как значительная часть времени уходит на накладные расходы (установление соединения, отправка заголовков и т.д.).
- По мере увеличения размера сообщения пропускная способность возрастает и достигает максимума для сообщений размером несколько мегабайт.
- Для очень больших сообщений (16 МБ) пропускная способность может немного снижаться из-за ограничений системы.

## Анализ графиков

Для визуализации результатов экспериментов был создан график с использованием gnuplot, который показывает два важных аспекта производительности:

### График 1: Пропускная способность

Пропускная способность и передача данных - это связанные, но различные понятия:
- Передача данных - это процесс перемещения информации от одного узла к другому
- Пропускная способность - это максимальная скорость, с которой данные могут быть переданы по каналу связи (измеряется в МБ/с)

Пропускная способность показывает, насколько быстро данные могут передаваться между процессами. Высокая пропускная способность означает, что система может эффективно передавать большие объемы данных за короткое время. Это критично для приложений, работающих с большими массивами данных, потому что:

1. Такие приложения часто требуют синхронизации между процессами, которые работают с общими наборами данных
2. Приложения машинного обучения, научные симуляции и базы данных часто обмениваются большими объемами данных между узлами
3. Низкая пропускная способность становится узким местом, ограничивающим общую производительность системы
4. Чем выше пропускная способность, тем меньше времени уходит на коммуникации и тем больше времени система может тратить на вычисления

Первый график (верхний) демонстрирует зависимость пропускной способности от размера сообщения. На этом графике:

- По оси X отложен размер сообщения в байтах (логарифмическая шкала)
- По оси Y отложена пропускная способность в МБ/с (линейная шкала)
- Использованы три цветовые кривые для обозначения разных уровней:
  - Зеленая линия (#2E7D32): Уровень 1 (Контроллер памяти)
  - Оранжевая линия (#F57C00): Уровень 2 (Внутрисистемная шина)
  - Красная линия (#C628): Уровень 3 (Сетевой адаптер)

На графике видно, что:
1. Пропускная способность всех уровней возрастает с увеличением размера сообщения до определенного порога (обычно до нескольких мегабайт), после которого начинает немного снижаться
2. Уровень 1 (контроллер памяти) демонстрирует наивысшую пропускную способность, достигающую ~16 ГБ/с для больших сообщений
3. Уровень 2 (внутрисистемная шина) имеет промежуточные характеристики, с пропускной способностью до ~10 ГБ/с
4. Уровень 3 (сетевой адаптер) показывает наихудшие результаты, с пропускной способностью около 140 МБ/с

Эти данные говорят нам о том, что при увеличении размера сообщений пропускная способность сначала возрастает, достигая пика при оптимальном размере сообщения, а затем может немного снижаться. Это объясняется тем, что при малых размерах сообщений значительная часть времени уходит на накладные расходы (установление соединения, обработка заголовков), а при слишком больших размерах могут возникать ограничения, связанные с буферизацией и обработкой данных.

### График 2: Время передачи (латентность)

4. **Использование асинхронных операций**: Асинхронные операции MPI позволяют скрывать задержки коммуникаций за счет перекрытия передачи данных с вычислениями, что особенно важно на более медленных уровнях коммуникаций.

5. **Важность измерений**: Эмпирические измерения производительности необходимы для понимания реального поведения системы и принятия обоснованных решений при оптимизации приложений.
